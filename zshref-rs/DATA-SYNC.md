# Cross-repo data-sync design — TS → Rust

> **Status.** Landed as of the option-6 dual-mode build (see `zshref-rs/build.rs` and `corpus.rs`). Document retained as the design record: why this shape, what the trade-offs were, and what the extraction-day cleanup looks like. Describes how the Rust crate `zshref` consumes JSON artifacts generated by the TypeScript `zsh-core` + `zsh-core-tooldef` packages, across the monorepo phase and the post-extraction phase (when `zshref-rs/` becomes its own repo).

---

## Problem shape

The Rust binary embeds ~940 KB of JSON — the zsh reference corpus plus
the MCP tool definitions — via `include_bytes!`. Those JSONs are
computed by a deterministic TS pipeline (parse → render → emit). They
are **not** source code, they are **generated data**, and they flow
one-way: TS produces them, everything else consumes them.

In the monorepo, Rust reaches across workspace boundaries — the original literal form being:

```rust
// zshref-rs/src/corpus.rs — original (pre-option-6)
const INDEX_JSON: &[u8] =
    include_bytes!("../../packages/zsh-core/dist/json/index.json");
```

After extraction, that relative path no longer exists: the two repos are siblings, not parent-and-child. Something has to bridge the gap. (Option 6 below replaces the literal with a `cfg`-gated macro that routes the same compile-time embed to either the monorepo path or a vendored `data/` dir.)

Two lifecycles, each with its own demands:

- **Monorepo (today).** `make cli` runs `pnpm build` first, then
  `cargo build`. Paths are relative; "latest TS source" is always
  present in the tree. Wonderful for iteration; irrelevant at release
  time.
- **Post-extraction.** Users run `cargo install zshref`. They have rust
  and nothing else — no Node, no pnpm, no checkout of the TS repo.
  Data must be inside the published `.crate`, already rendered.

The design has to be sane at both ends and at every point between them.

## Two views, both legitimate

Your sketch was correct. There are two fundamentally different framings
of "Rust depends on TS-produced JSON":

**View A — source dependency.** Rust depends on TS *source code* at
some pinned revision. Whoever builds the Rust binary is responsible for
regenerating the JSONs from that source.

**View B — artifact dependency.** Rust depends on TS *build output* at
some version. The regeneration happened elsewhere; Rust just consumes
the bytes.

View A is what a monorepo gives you for free. View B is what a package
ecosystem gives you for free. The question is which view — or which
blend — survives extraction, and what the cost of each blend is.

## The options, concretely

### 1. Vendor the JSONs into the Rust repo (always checked in)

`zshref-rs/data/*.json` is tracked by git. A sync script (pre- or post-
extraction) copies fresh TS-generated files into it. `include_bytes!`
reads from `data/`.

- **End-user install.** Trivially offline, rust-only. `cargo install
  zshref` just works.
- **Monorepo dev.** Every TS-source change that affects data has to be
  followed by `make vendor`, committed. Two commits per logical
  change. CI enforces "`make vendor && git diff --exit-code`" to
  prevent drift.
- **Post-extraction.** The Rust repo self-contains everything it needs.
  Sync script pivots from "copy from sibling package" to "clone TS repo
  at pinned ref, run `pnpm build`, copy".
- **Cost.** Generated files in git. You flagged this as a hard-avoid;
  it is mildly attenuated here because the vendored data comes from a
  *different repo* post-extraction, but during the monorepo phase the
  data is sibling-package output landing in a sibling crate's tree.
  That is "in the same repo" in spirit.

### 2. Fetch JSONs at `cargo build` time from a URL

`build.rs` downloads a release blob from GitHub and writes it to
`OUT_DIR`. `include_bytes!` reads from `OUT_DIR`.

- **End-user install.** Requires network every build. Breaks inside
  corporate proxies, sandboxed CI, air-gapped environments, Nix/Guix
  builds.
- **Security.** Moves the supply-chain trust boundary to a GitHub
  release + CDN. Anyone who can mutate that artifact can silently
  trojan every subsequent `cargo install zshref`.
- **crates.io etiquette.** Network access in build scripts is formally
  allowed but community- and policy-discouraged. Many downstream
  consumers (security auditors, `cargo vet`, reproducible-build shops)
  will reject it.
- **Verdict.** Dead-end.

### 3. Git submodule of the TS repo; `build.rs` runs `pnpm build`

`zshref-rs/vendor/ts` is a submodule pinned to a commit; `build.rs`
invokes `pnpm install && pnpm build` in it and reads the fresh output.

- **End-user install.** Requires Node + pnpm on the user's build
  machine, a multi-minute first build, submodule dance during clone.
  `cargo install zshref` stops being a 10-second experience.
- **Verdict.** Dead-end.

### 4. Separate `zshref-data` data-only crate on crates.io

Split the data into its own publishable crate. `zshref` depends on
`zshref-data = "x.y"`. The data crate's release cadence tracks TS-data
changes; the CLI crate's cadence tracks CLI changes.

- **Pros.** Version decoupling. Reusable by other Rust consumers.
- **Cons.** Second crate to publish; more complexity with no current
  demand (no other Rust consumer exists). Doesn't solve the "who runs
  the TS build" problem — it moves.
- **Verdict.** Defer. Worth revisiting if another Rust consumer
  appears.

### 5. Gitignored `data/`, populated on demand, always local

`zshref-rs/data/*.json` is in the tree but gitignored. A small make
target fills it from the TS workspace. `include_bytes!` reads from
`data/`. `cargo publish` uses `include = [..., "data/**"]` so the
published tarball ships with populated `data/`; git never tracks it.

- **Pros.** No checked-in generated files. Package is self-contained.
  Offline-safe at install time.
- **Cons.** `git clone && cargo build` fails until the user runs
  `make vendor`. Sharp edge for new contributors. The
  `include`-via-gitignore pattern is legal but non-obvious.
- **Verdict.** Workable, but pre-extraction-only half of (6).

### 6. Hybrid dual-mode with auto-detecting `build.rs` — **recommended**

Combine the best half of (1) and (5): read from the **monorepo path**
when it exists, read from the **vendored `data/` dir** otherwise. The
build script picks the mode; `include_bytes!` uses a macro-gated path.
Published `.crate` files ship with `data/` populated → vendored mode
kicks in automatically at install time.

Pre-release today, nothing changes for dev: the monorepo path is
auto-detected, builds proceed as now, no `data/` dir needed.

Pre-release at publish time (if we cut a test release from the
monorepo), `make vendor` populates `data/` and `cargo publish` ships
it.

Post-extraction, the monorepo branch of the auto-detection is unused;
`data/` is committed; `build.rs` can drop the dead branch.

- **Pros.** Today's ergonomics preserved. Vendored path validated by
  CI today, long before extraction. End-user install is toolchain-free.
  No network at build time. Post-extraction requires only trivial
  cleanup.
- **Cons.** Two code paths in `build.rs` / `corpus.rs` during the
  monorepo phase. Cost: ~30 lines of conditional compilation.
  Entirely manageable; the benefit — testable end-to-end publish path
  *before* we ever extract — is large.

## Recommendation: option 6

```text
         ┌─────────────────────────────────┐
         │  build.rs auto-detect           │
         │                                 │
         │  data/ exists     →  "vendored" │
         │  ../packages/…    →  "monorepo" │
         │  neither          →  panic!     │
         └──────────────┬──────────────────┘
                        │
                        ▼
            cargo:rustc-cfg=data_source="…"
                        │
                        ▼
   #[cfg(data_source = "vendored")]   include_bytes!("../data/index.json")
   #[cfg(data_source = "monorepo")]   include_bytes!("../../packages/…/index.json")
```

### How it works

**`zshref-rs/build.rs`** (new):

```rust
use std::{env, path::Path};

fn main() {
    let manifest = env::var("CARGO_MANIFEST_DIR").unwrap();
    let manifest = Path::new(&manifest);

    let vendored = manifest.join("data/index.json").exists();
    let monorepo = manifest
        .join("../packages/zsh-core/dist/json/index.json")
        .exists();

    let source = match (vendored, monorepo) {
        (true, _)     => "vendored",
        (false, true) => "monorepo",
        (false, false) => panic!(
            "No data source found. Expected either:\n\
             - {manifest}/data/*.json   (vendored mode; run `make vendor`)\n\
             - ../packages/.../dist/json/*.json (monorepo mode; run `make cli` from repo root)\n",
            manifest = manifest.display()
        ),
    };

    println!("cargo:rustc-check-cfg=cfg(data_source, values(\"vendored\", \"monorepo\"))");
    println!("cargo:rustc-cfg=data_source=\"{source}\"");

    println!("cargo:rerun-if-changed=data");
    println!("cargo:rerun-if-changed=../packages/zsh-core/dist/json");
    println!("cargo:rerun-if-changed=../packages/zsh-core-tooldef/dist/json");
}
```

**`zshref-rs/src/corpus.rs`** (replace the three `include_bytes!`
blocks with path macros):

```rust
#[cfg(data_source = "vendored")]
macro_rules! corpus_path  { ($f:literal) => { concat!("../data/", $f) }; }
#[cfg(data_source = "vendored")]
macro_rules! tooldef_path { ($f:literal) => { concat!("../data/", $f) }; }

#[cfg(data_source = "monorepo")]
macro_rules! corpus_path  { ($f:literal) => { concat!("../../packages/zsh-core/dist/json/", $f) }; }
#[cfg(data_source = "monorepo")]
macro_rules! tooldef_path { ($f:literal) => { concat!("../../packages/zsh-core-tooldef/dist/json/", $f) }; }

const TOOLDEF_JSON: &[u8] = include_bytes!(tooldef_path!("tooldef.json"));
const INDEX_JSON:   &[u8] = include_bytes!(corpus_path!("index.json"));

macro_rules! include_category {
    ($cat:literal, $file:literal) => {
        ($cat, include_bytes!(corpus_path!($file)) as &[u8])
    };
}
```

**Repo-root `.gitignore`** (one line):

```
zshref-rs/data/
```

(A dedicated `zshref-rs/.gitignore` isn't created — the root file covers it pre-extraction. Post-extraction, when `data/` is committed, the entry comes out.)

**Root `Makefile`** adds:

```make
.PHONY: vendor vendor-clean cli-vendored cli-vendored-test

vendor: artifacts
	@mkdir -p zshref-rs/data
	@cp packages/zsh-core/dist/json/*.json zshref-rs/data/
	@cp packages/zsh-core-tooldef/dist/json/tooldef.json zshref-rs/data/
	@echo "vendored → zshref-rs/data/"

vendor-clean:
	rm -rf zshref-rs/data

cli-vendored: vendor-clean vendor
	cd zshref-rs && cargo build --release

cli-vendored-test: vendor-clean vendor
	cd zshref-rs && cargo test
```

`vendor-clean` runs first so a stale `data/` never satisfies the
auto-detection incorrectly.

**`zshref-rs/Cargo.toml`** — add (for when `publish = true`):

```toml
include = [
    "src/**/*.rs",
    "build.rs",
    "Cargo.toml",
    "Cargo.lock",
    "README.md",
    "LICENSE",
    "THIRD_PARTY_NOTICES.md",
    "data/*.json",
]
```

Gitignored `data/` gets *included* in the published `.crate` because
`include` overrides `.gitignore` for the tarball. That's the whole
trick.

**`.github/workflows/ci-rust.yml`** — the existing `rust` job gained one extra step after the monorepo-mode fmt+clippy+test run:

```yaml
- name: Vendored-mode test + package dry-run
  run: |
    make cli-vendored-test
    make cli-package
```

`make cli-package` runs `cargo package --allow-dirty`, which builds the publishable tarball and verifies it compiles standalone. That's the real prize: it proves the published crate **would build standalone** with zero outside deps beyond rustc. If this step is green today, it's green on any `cargo publish`.

### Why the auto-detection, not a feature flag?

A feature flag (`--features vendored`) is the obvious alternative. The
problem: `cargo install` defaults. The published crate would need
`default-features = ["vendored"]`, and the monorepo dev path would
need someone to remember `--no-default-features`. Too easy to get
wrong.

`build.rs` detection is zero-overhead and invisible: whichever data is
present wins. The mode is logged (`println!` via `cargo:warning=…`
if we want it loud); no one has to remember a flag.

### What flips at extraction

- Delete the `monorepo` branch in `build.rs` and the corresponding
  macro arm in `corpus.rs`. (Three-line change.)
- Commit `data/` (remove the `/data/` line from `.gitignore`).
- The Makefile shrinks: `artifacts` target goes away; `vendor` becomes
  a sync script that clones the TS repo at a pinned commit (stored in
  a `DATA_COMMIT` file or similar) and runs its build.
- The monorepo-only CI job disappears.

Roughly 50 lines of cleanup on extraction day.

## Testability matrix

|                         | monorepo mode | vendored mode | extraction simulation |
| ----------------------- | :-----------: | :-----------: | :-------------------: |
| `make cli`              | ✓             |               |                       |
| `make cli-test`         | ✓             |               |                       |
| `make cli-vendored`     |               | ✓             |                       |
| `make cli-vendored-test`|               | ✓             |                       |
| `make cli-package`      |               | ✓             | ✓                     |

`make cli-package` (which is `cargo package --allow-dirty` against
`zshref-rs/`) is the extraction simulation: cargo unpacks the tarball
into `target/package/zshref-<version>/` and compiles it *standalone*
— no monorepo visible. If that build passes, extraction-day `cargo
build` will too.

## Drift guarantees

Two kinds of drift to worry about:

1. **TS source → vendored `data/`.** Anyone who edits TS source but
   forgets to re-vendor ships stale data on next publish. Guard: the
   `rust-vendored` CI job does a clean `make vendor-clean && make
   vendor && make cli-vendored-test`. Drift from stale vendored data
   would surface as: either the build embeds different bytes in the
   two modes (CI compares `zshref info` output), or the drift-guard
   `#[cfg(test)]` tests in `corpus.rs` fail.

2. **Schema drift (Rust structs vs. TS JSON shape).** Already handled
   by the four drift-guard tests in `corpus.rs::tests`. Unchanged by
   this design.

## Answers to your sketch

> *"it depends on a snapshot of the checked-in ts files"*

Yes — and the Rust build does not have to kick off JSON generation
itself. The generation runs on the *producer* side (pre-extraction:
the monorepo's `make artifacts`; post-extraction: the TS repo's own
CI), and the producer hands Rust a ready blob. Rust never runs TS
code. That's important for `cargo install zshref` to remain a Rust-
only experience.

> *"it depends on build artifacts from some other repo"*

Yes, post-extraction this is literally the case — but the artifacts
don't have to come from a GitHub release fetch at build time. They
come over as *source-of-the-Rust-crate* at pre-publish time (via
`make vendor` (post-extraction: whatever sync mechanism replaces it)), then ride inside the published `.crate`
tarball forever. The user machine only ever sees local files.

The blend of your two framings — source dep *from the producer's
perspective*, artifact dep *from the consumer's perspective*, with a
sync script straddling the boundary — is what Option 6 formalizes.

## Open questions

- **Sync trigger discipline post-extraction.** When TS data changes
  upstream, how does the Rust repo learn? Options: (a) manual — the
  person bumping the TS version opens a PR on the Rust repo bumping
  `DATA_COMMIT`; (b) a GitHub action on the Rust repo that polls the
  TS repo weekly and opens a PR if data changed. (b) is the classic
  "dependabot for vendored data" pattern. Decide on extraction day.
- **Does the Rust repo track one data version, or a range?** Simplest:
  one. `DATA_COMMIT` (or `DATA_VERSION`, once TS gets tags) is a pin.
  The Rust repo releases when the pin bumps, not on TS's cadence.
- **Cargo.lock policy post-extraction.** Standard for a bin crate
  (commit it), but worth noting in the extraction doc.
- **Size budget.** 940 KB is fine baked into a binary. If corpus grows
  past, say, 5 MB, reconsider: either compress (`zstd`/`brotli` at
  build time, decompress at load) or split the data crate (option 4).
